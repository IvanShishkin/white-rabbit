# White Rabbit — дизайн-спека

**Дата:** 2026-06-30
**Статус:** дизайн утверждён, готов к планированию реализации
**Тип проекта:** отдельный git-репозиторий + Claude Code plugin

---

## 1. Vision

**White Rabbit** — персональный security-копайлот для аудита собственной инфраструктуры,
работающий внутри Claude Code. Он не сканер: «движки» (если нужны) берутся готовыми, а
ценность инструмента — в **оркестрации, приоритизации по реальной эксплуатируемости и
AI-триаже ложных срабатываний**.

Сырой сканер выдаёт 500 «уязвимостей». White Rabbit должен сказать:
> «Вот эти 3 чините сегодня, остальное — шум, и вот почему.»

Имя — двойной смысл: **white**-box аудит + *«follow the white rabbit»* (идти по следу,
вскрывать скрытое в логах и коде; показать реальную глубину «норы»).

### Целевое применение
Личная инфраструктура: проекты в разработке и боевые серверы. Запускается локально
в Claude Code; до сервера дотягивается по SSH.

---

## 2. Тип аудита и поста (posture)

- **White-box (authenticated) аудит собственной инфраструктуры.** Полный доступ к коду,
  конфигам, логам, списку пакетов — смотрим «изнутри», видим первопричину, а не симптом.
- **Строго read-only by construction.** Инструмент **ничего не меняет**. Фиксы выдаются
  как готовые команды, которые пользователь запускает сам. Это убирает целый класс рисков
  при работе на боевых серверах.
- **Активная эксплуатация — вне скоупа.** Никакого брутфорса, запуска эксплойтов,
  интрузивного пентеста. Это сознательная граница (безопасность + легальность).
- **Опционально — пассивная outside-in проверка периметра** (порты / TLS / security-заголовки /
  истечение сертов): «что видит атакующий снаружи», без атаки.

## 3. Архитектура

### Главный принцип: расширение мозга = markdown, а не код

Мозг инструмента — **сам Claude Code**. White Rabbit оформлен как **Claude Code plugin**.
Новая возможность аудита добавляется **декларативно** — новым плейбуком/справочником в
markdown, а не новым сканером.

### Путь A — zero-install
Claude ходит по SSH и собирает данные **нативными командами**, которые уже есть на хосте
(`ss`, `journalctl`, `dpkg`, `ufw status`, чтение конфигов). На сервер **ничего не ставится** —
работает на любом хосте, к которому есть доступ. Тяжёлые сканеры (Lynis/Trivy/gitleaks) —
возможное расширение в будущих фазах по принципу «есть — использую, нет — обхожусь нативом».

### Слои

1. **Плейбуки (`skills/`) — это и есть «мозг».**
   Каждый домен аудита = скилл-методичка: что проверять, какими командами собрать, как
   интерпретировать, как оценить серьёзность. *Расширяешь мозг — дописываешь сюда.*

2. **Коллекторы (`scripts/collect/`) — детерминированный сбор.**
   Claude по SSH запускает один скрипт (например `server_snapshot.sh user@host`) и получает
   весь дамп состояния за один заход — быстрее и воспроизводимее, чем 30 отдельных команд.
   Скрипты содержат только read-only команды → сами по себе гвардрейл.

3. **Рассуждение (сам Claude).**
   Дамп + правила интерпретации из плейбука + рубрика риск-скоринга → триажированные находки
   (отсекает шум, объясняет «почему важно»).

4. **Оркестрация (`skills/wr-orchestrator/`).**
   Раздаёт работу субагентам (по одному на домен) параллельно (fan-out, ложится на стек Orca) →
   собирает находки → дедуп/скоринг → единый отчёт.

5. **Память/бейзлайн (`reports/`).**
   Хранит прошлые снимки → дрейф и дельта («что изменилось с прошлой проверки», «новые находки»).

### Структура репозитория

```

├── commands/

├── skills/                    # ← МОЗГ. расширяешь здесь
│   ├── wr-orchestrator/SKILL.md   # fan-out + синтез отчёта
│   ├── wr-server-audit/SKILL.md   # MVP: хардненинг хоста
│   ├── wr-log-hunt/SKILL.md       # MVP: разбор логов на атаки
│   ├── wr-code-audit/SKILL.md     # фаза 2
│   └── wr-deps-cve/SKILL.md       # фаза 3
├── agents/
│   └── wr-collector.md        # субагент-сборщик для параллельного fan-out
├── scripts/
│   ├── collect/               # bash-коллекторы (один SSH-заход, read-only)
│   │   ├── server_snapshot.sh
│   │   └── log_pull.sh
│   └── parse/                 # python-нормализаторы → findings.json
│       └── normalize.py
├── knowledge/                 # справочники, на которые ссылаются плейбуки
│   ├── checks/                # каталог проверок по доменам (CIS-style)
│   │   ├── ssh.md
│   │   ├── firewall.md
│   │   └── kernel.md
│   ├── scoring.md             # формула риска (CVSS×EPSS×KEV×критичность)
│   └── attack-patterns.md     # маппинг логов → MITRE ATT&CK
├── policy/                    # ← ГВАРДРЕЙЛСЫ (декларация)
│   ├── guardrails.md          # политика — источник правды
│   ├── allowed-commands.txt   # белый список read-only команд
│   └── denied-patterns.txt    # rm, kill, >, install, flush, stop ...
├── hooks/                     # ← ГВАРДРЕЙЛСЫ (enforcement)
│   ├── hooks.json             # регистрирует PreToolUse-гвард
│   └── guard.sh               # блокирует не-read-only команды
├── targets/
│   └── targets.example.yaml   # список серверов/профили (секреты в .gitignore)
├── reports/                   # вывод: находки, бейзлайны, тренд
└── README.md
```

### Поток данных за один прогон (`/wr server`)

```
ты: /wr server  (или "прогони аудит VM")
   │
   ▼
wr-orchestrator  ──fan-out──▶ субагент: server-audit
   │                          │  1. ssh → server_snapshot.sh
   │                          │  2. читает дамп + knowledge/checks/*
   │                          │  3. Claude триажит → findings (server)
   │              ──fan-out──▶ субагент: log-hunt
   │                          │  ssh → log_pull.sh → паттерны атак → findings (logs)
   ▼
сбор находок → дедуп → риск-скоринг (scoring.md) → сравнение с reports/ (дельта)
   │
   ▼
единый отчёт: «N критичных, чините сегодня» + команды-фиксы → Slack/файл
```

## 4. Гвардрейлсы (безопасность самого агента)

Различаем **«посоветовал» (advisory)** и **«запретил по-настоящему» (enforced)**.

| Слой | Где | Тип | Что делает |
|------|-----|-----|------------|
| Политика | `policy/guardrails.md` | декларация | единый источник правды: read-only, без эксплуатации, секреты не в отчёты, фиксы только как подсказки |
| **Хуки** | `hooks/guard.sh` + `hooks.json` | **жёсткий enforcement** | `PreToolUse` перехватывает каждую Bash/SSH-команду до выполнения и **блокирует** мутации (`rm`, `kill`, `>`/`>>`, `iptables -F`, `apt install`, `systemctl stop`...). Enforce-ит харнесс, не добрая воля модели |
| Permissions | `.claude-plugin/plugin.json` | жёсткий | allow/deny-лист инструментов и паттернов команд |
| Плейбуки | `skills/*/SKILL.md` (operating principles) | мягкий | формируют намерение; сами по себе не считаются гвардрейлом |
| Коллекторы | `scripts/collect/*.sh` | by construction | отревьюенные read-only скрипты вместо ad-hoc команд (defense in depth) |

**Поста: строго read-only.** Хук запрещает любую мутацию в принципе — логика «подтверждения
изменений» не нужна. Секреты (`.env`, ключи) никогда не попадают в отчёты/логи.

## 5. Возможности (полный vision)

> Все четыре подсистемы строятся на одном принципе: приоритизация по реальной
> эксплуатируемости + AI-триаж ложных срабатываний.

### 5.1 Аудит кода *(фаза 2)*
- SAST + поиск секретов + линтеры по стеку (PHP/Python/Go).
- Сканирование **всей git-истории** (утёкший в старом коммите ключ всё ещё валиден).
- **Валидация секретов** — живой ключ vs «похоже на ключ».
- **Diff-aware** режим — только изменённое на коммит/PR.
- **Baseline + дельта** — алерт только на новые находки.
- **AI-триаж**: Claude читает код вокруг находки (тест/мок → игнор; прод на необработанном
  вводе → критично). Главный дифференциатор.
- Маппинг на **OWASP Top 10 / CWE**.

### 5.2 Версии ПО и CVE *(фаза 3)*
- **SBOM** (CycloneDX/SPDX) как артефакт; затем скан по базам CVE.
- **Reachability-анализ** — уязвимая функция реально вызывается из кода (режет 60–80% шума).
- **Приоритизация по EPSS + CISA KEV**, а не только CVSS.
- **VEX** — помечать неэксплуатируемые CVE как заглушенные с обоснованием.
- Бонус: лицензионный аудит из SBOM.

### 5.3 Аудит и хардненинг сервера *(MVP)*
- Карта поверхности атаки: открытые порты (`ss`), слушающие сервисы.
- Аудит конфигов: SSH (root-login, парольный вход, ключи), sudo, firewall (ufw/iptables/nftables),
  cron, пользователи и права.
- Patch level ядра/ОС — необновлённые критичные пакеты.
- TLS/сертификаты: слабые шифры/протоколы, **дата истечения сертов**, security-заголовки.
- Drift-детект: сравнение с прошлым снимком — «что изменилось».
- *(расширение)* контроль целостности файлов (AIDE), поиск руткитов (rkhunter), IaC-аудит.

### 5.4 Анализ логов / детект вторжений *(MVP)*
- Детект паттернов атак: брутфорс SSH, перебор веб-форм, SQLi/XSS/path-traversal в URL, сканеры.
- **GeoIP + threat-intel обогащение IP** (AbuseIPDB / CrowdSec CTI / Spamhaus).
- Корреляция и аномалии: всплеск 401/403, необычное время/гео входа.
- **Маппинг на MITRE ATT&CK**.
- Сводка «кого забанил fail2ban/CrowdSec и почему».

## 6. Сквозные слои

- **Нормализация в единый формат** (SARIF-совместимый) → дедуп, единый отчёт, история.
- **Движок риск-скоринга**: `CVSS × EPSS × KEV × reachability × критичность актива` →
  один отсортированный список «чините это первым».
- **AI-триаж** — отдельный слой, фильтрует ложные срабатывания с учётом контекста.
- **Отчётность**: единый MD/HTML-отчёт + тренд во времени + конкретные команды-фиксы.
  Доставка в **Slack** (переиспользуем существующий MCP-паттерн рассылки).
- **Расписание**: еженедельный security-дайджест по cron/systemd на VM.
- **Мульти-агентный fan-out**: подсистемы гоняются параллельно (стек Orca).

## 7. MVP

**Ядро MVP = Сервер + Логи** + сквозные слои (нормализация → риск-скоринг → AI-триаж →
отчёт в Slack).

Обоснование выбора:
- Меньше всего пересекается с уже имеющимся встроенным `/security-review` (он про код).
- Путь A (zero-install, нативные команды) раскрывается именно на сервере.
- Прямое применение прямо сейчас — есть боевая VM.
- Сервер и логи естественно ходят парой: «как настроено» + «кто уже стучится».

### Definition of Done для MVP
1. Плагин ставится в Claude Code; команда `/wr server` и `/wr logs` работают.
2. `server_snapshot.sh` и `log_pull.sh` собирают данные по SSH одним заходом (read-only).
3. Гвардрейл-хук блокирует любую мутирующую команду (проверено тестом).
4. Оркестратор выдаёт единый отчёт с риск-скорингом и дельтой против прошлого прогона.
5. Отчёт доставляется в Slack.
6. Прогон на боевой VM (`hola@111.88.146.83`) даёт осмысленный приоритизированный результат.

## 8. Roadmap (фазы)

- **Фаза 1 (MVP):** Сервер + Логи + сквозные слои + гвардрейлсы + Slack-доставка.
- **Фаза 2:** Аудит кода (SAST/секреты/git-история, AI-триаж, OWASP/CWE).
- **Фаза 3:** Версии/CVE (SBOM, reachability, EPSS/KEV, VEX).
- **Фаза 4:** Расширения — пассивный outside-in периметр, тяжёлые сканеры по опции,
  AIDE/rkhunter, IaC-аудит, авто-PR на безопасные апдейты.

---

## 9. Вне скоупа (YAGNI)

- Активная эксплуатация / пентест / брутфорс.
- Любые мутации инфраструктуры (только подсказки-команды).
- Установка агентов/демонов на сервер (zero-install, путь A).
- Полноценный SIEM/веб-консоль (это не Wazuh).
- Собственные сканеры — используем готовые/нативные.

## 11. Открытые вопросы

- Имя CLI/скилла при вызове: `/wr` подтверждено как короткая форма; полное имя
  `white-rabbit` оставляем как имя плагина.
- Сервер: проверка по SSH удалённо подтверждена (путь A). Режим «запуск прямо на хосте» —
  не требуется для MVP.
- Точное место репозитория в `~/Projects/` и регистрация в `projects.yaml` (через
  `project-scan`) — на этапе создания репо.
